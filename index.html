用户服务协议

  • 您传递给的 React 树hydrateRoot需要产生与服务器上相同的输出。

    这对用户体验很重要。在您的 JavaScript 代码加载之前，用户会花一些时间查看服务器生成的 HTML。服务器渲染通过显示其输出的 HTML 快照创建应用程序加载更快的错觉。突然显示不同的内容打破了这种幻想。这就是为什么在水合作用期间服务器渲染输出必须与客户端上的初始渲染输出相匹配。 导致水合作用错误的最常见原因包括： 根节点内 React 生成的 HTML 周围的额外空格（如换行符）。 typeof window !== 'undefined'在您的渲染逻辑中使用检查。 window.matchMedia在您的渲染逻辑中使用仅限浏览器的 API 。 在服务器和客户端上呈现不同的数据。 React 可以从一些水合作用错误中恢复，但你必须像修复其他错误一样修复它们。在最好的情况下，它们会导致应用程序变慢；在最坏的情况下，事件处理程序会附加到错误的元素。

  • root 完成 hydrating 后，你可以调用root.render更新 root React

    组件。与 不同createRoot，您通常不需要这样做，因为初始内容已经呈现为 HTML。 如果你root.render在 hydration 之后的某个时刻调用，并且组件树结构与之前渲染的相匹配，React 将保留状态。请注意您如何输入输入，这意味着render在此示例中每秒重复调用的更新不是破坏性的： index.html index.js

  • 参数 domNode：在服务器上呈现为根元素的DOM 元素。

    reactNode：用于呈现现有 HTML 的“React 节点”。这通常是一段 JSX ，类似于 使用.ReactDOM ServerrenderToPipeableStream optional options：一个对象包含此 React 根的选项。 onRecoverableError：当 React 自动从错误中恢复时调用的可选回调。 identifierPrefix: 可选前缀 React 用于useId. 在同一页面上使用多个根目录时，有助于避免冲突。必须与服务器上使用的前缀相同。 退货 hydrateRoot返回具有两个方法的对象：render和unmount。 注意事项 hydrateRoot()期望呈现的内容与服务器呈现的内容相同。您应该将不匹配视为错误并修复它们。 在开发模式下，React 会在水合作用期间警告不匹配。不保证在不匹配的情况下会修补属性差异。出于性能原因，这很重要，因为在大多数应用程序中，不匹配的情况很少见，因此验证所有标记的成本非常高。 您的应用程序中可能只有一次hydrateRoot调用。如果您使用框架，它可能会为您执行此调用。 如果您的应用程序是客户端呈现的，但尚未呈现 HTML，hydrateRoot()则不支持使用。改用createRoot()。
